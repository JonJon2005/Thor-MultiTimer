ANDROID APP PROJECT CONTEXT REPORT

PROJECT OVERVIEW
- Project name: Thor MultiTimer
- Current package name: com.example.thormultitimer
- Current save location: /path/to/thor-multitimer
- This is a Kotlin + Jetpack Compose Android app created in Android Studio.
- The project started as the default Empty Activity template but is now a custom multi-timer app.
- The app is intentionally being built in a beginner-friendly way:
  - Kotlin only
  - Jetpack Compose only
  - Material 3
  - Gradle Kotlin DSL
  - no legacy XML view UI
- The user is learning Android development while building this app, so future work should stay explainable and incremental.

PRIMARY GOAL
- Build a normal Android app intended to be used ONLY on the bottom screen of the AYN Thor.
- Treat the target as a single compact Android screen, not as a dual-screen or foldable app.
- Do NOT introduce:
  - dual-screen APIs
  - spanning
  - foldable APIs
  - second-display-specific architecture
  - Bluetooth controller assumptions
- Design the app around:
  - fast interactions
  - no scroll-heavy navigation
  - touch-friendly controls
  - controller / D-pad support for Thor hardware buttons

CURRENT APP CONCEPT
- The app concept is now clearly defined:
  - It is a multi-timer app.
  - The user can quickly create and manage multiple timers at once.
  - The UI is optimized for a compact device screen and hardware button input.
- The current app is not just a prototype shell anymore.
- It contains a working dashboard, timer logic, settings, theme switching, accent switching, alert behavior, haptics, and hardware controller navigation.

HARDWARE TARGET / SCREEN TARGET
- Target device: AYN Thor bottom screen
- Target usage: bottom screen only
- Do not build for spanning across both displays
- Do not build special multi-display app logic unless explicitly requested later
- Correct bottom-screen resolution assumption:
  - Resolution: 1080 x 1240
  - Orientation assumption: portrait
  - Screen size reference: about 3.92 inches

  - This is a compact portrait display with less room than a normal phone
  - Oversized headers and deep vertical stacks can easily break the layout

  - The dashboard should avoid hidden content and unnecessary text

CURRENT CORE DATA MODEL
- TimerItem currently contains:
  - id
  - totalSeconds
  - remainingSeconds
  - isRunning
  - endTimeMillis
- Timer state is stored in:
  - mutableStateListOf<TimerItem>()
- Finished timer IDs are tracked separately so the finish alert popup can be shown and cleared safely.

CURRENT TIMER LOGIC
- addTimer(totalSeconds, startImmediately = false):
  - validates totalSeconds > 0
  - appends the new timer so existing card order stays stable
  - sends currentPage to the page that contains the new timer
  - optionally starts immediately by setting endTimeMillis
- updateTimer(timerId, transform):
  - finds the timer by id
  - replaces it with a transformed copy
- Countdown behavior:
  - driven by a LaunchedEffect loop
  - uses System.currentTimeMillis()
  - recomputes remaining seconds from endTimeMillis
  - stops timers when remaining time reaches 0
- Refresh cadence:
  - faster when at least one timer is running
  - slower when idle
- Pause behavior:
  - computes remaining time from endTimeMillis
  - clears running state
  - clears endTimeMillis
- Start behavior:
  - resumes from remainingSeconds if timer is paused
  - restarts from totalSeconds if remainingSeconds == 0
- Reset behavior:
  - restores remainingSeconds to totalSeconds
  - clears running state
  - clears endTimeMillis
- Remove behavior:
  - deletes the timer from the list

CURRENT MAIN SCREEN / DASHBOARD
- The main screen is a fixed dashboard layout.
- It is intentionally not a scrollable timer list.
- Main screen structure:
  - header row
  - quick action row
  - 2x2 timer grid
  - page controls only when needed
- Header:
  - title: Thor MultiTimer
  - settings gear icon on the top right
  - no timer count / running count text
- Quick action row:
  - Custom
  - 5m
  - 10m
  - 15m
- Quick preset behavior:
  - creates timer
  - starts timer immediately
  - controller focus stays on the preset button that created the timer
- Custom behavior:
  - opens custom timer dialog
  - created timers do not auto-start
  - controller focus stays on Custom after returning to the dashboard
- Page behavior:
  - 4 timers max visible per page
  - page controls appear only if timers exceed 4
  - Previous / Next switch pages
  - newly added timers appear in the next available card slot rather than displacing earlier timers

CURRENT TIMER CARD DESIGN
- Timer cards are shown in a 2x2 grid.
- Empty slots remain visible so the grid structure stays stable.
- Active / existing timer cards show:
  - large countdown
  - remove button in the top-right
  - Pause + Reset while running
  - Start or Restart while not running
- Empty slots show:
  - background only
  - outline only
  - no explanatory text
- Controller focus state is visually reflected on timer cards.
- Empty cards are intentionally styled to remain visible even in OLED mode.

CURRENT AUDIO / ALERT BEHAVIOR
- When a timer finishes:
  - the app attempts to play the device system alarm sound
  - if no default alarm sound exists, it falls back to the system notification sound
- This uses RingtoneManager rather than a bundled sound file.
- Emulator audio may be silent depending on emulator sound configuration.
- Real devices are more reliable for verifying audio behavior.

CURRENT HAPTIC / FEEDBACK BEHAVIOR
- Touch haptics:
  - main shared app buttons vibrate on tap
  - accent swatch buttons vibrate on tap
  - highlight swatch buttons vibrate on tap
- Controller haptics:
  - controller A-button confirmation now triggers haptic feedback too
  - controller focus movement uses a lighter haptic pulse
  - controller Y-close on dialogs triggers a dismiss haptic pulse
- Button visuals:
  - buttons use a press-scale animation for tap feedback

CURRENT CONTROLLER SUPPORT
- The app now supports hardware D-pad and face-button input for the Thor.
- This is not based on Bluetooth controller support.
- Controller support is handled explicitly in Compose key event processing.
- Supported conceptual inputs:
  - D-pad left
  - D-pad right
  - D-pad up
  - D-pad down
  - primary action (A / center / enter)
  - back action (B)
  - secondary action (X)
  - tertiary action (Y)
- Dashboard controller behavior:
  - D-pad moves focus logically across:
    - settings
    - custom
    - presets
    - timer cards
    - page controls
  - A / primary:
    - activates the focused item
    - starts/pauses timer cards
    - opens empty slot into timer creation flow
  - X / secondary on focused timer card:
    - reset timer
  - Y / tertiary on focused timer card:
    - remove timer
  - controller hint bar appears at the bottom only while controller input is active
  - controller mode times out after 10 seconds of inactivity
  - when controller mode times out:
    - the hint bar disappears
    - the visible controller focus ring disappears
    - the timer grid smoothly expands back into the freed space
- Dialog controller behavior:
  - D-pad moves between logical controls
  - A confirms the focused control
  - Y closes settings and custom timer panels
  - B does not close settings and custom timer panels
  - settings and custom timer dialogs show a top-right Y Close indicator
- Finish alert controller behavior:
  - A stops alert
  - B also stops alert
- Controller focus state uses a single animated ring that:
  - moves smoothly between focus targets
  - resizes to different button sizes
  - adapts to different rounded shapes
  - uses the selected highlight color

IMPORTANT INTERACTION CONSTRAINTS
- The user explicitly does not want behavior tied to Android hardware back/home navigation as the app’s normal flow.
- The app currently does not rely on hardware back/home behavior for user-visible core actions.
- Home cannot be overridden in a normal Android app anyway.
- Dialog dismissal and controller cancellation use explicit app logic.
- The main app should remain operable through on-screen controls and controller input without needing Android navigation buttons.

CURRENT LIMITATIONS / KNOWN GAPS
- Timers are still in-memory only.
- If the app process dies, timers are lost.
- Settings are persisted, but timers are not.
- Timers do not survive reboot or background process loss.
- There is no foreground service or background alarm scheduling.
- There are no Android notifications for finished timers.
- There is no vibration pattern customization yet.
- Controller mapping assumes standard Android key codes; Thor-specific remapping may be needed if the hardware reports unusual codes.
- There is no automated test coverage yet.

SAFE ASSUMPTIONS FOR FUTURE WORK
- Keep using Compose.
- Keep using a single-screen dashboard-first design.
- Avoid introducing scrolling as the main way to manage timers unless the user explicitly changes direction.
- Keep controls large and compact-screen friendly.
